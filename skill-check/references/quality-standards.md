# Skill 质量标准详解（判定细则 + 正反面例子）

本文件是 skill-check 的判定依据，即「六要素 · 六步骤」审核模型的标准全文：**六要素是审核标准（缺一不可），六步骤是审核流程**。核对被审 skill 时逐项对照，判定 ✅（达标）/ ⚠️（有但不足）/ ❌（缺失或错误）。所有示例均为通用虚构示例，仅用于说明标准。

---

## 一、选题标准（值不值得打包）

**该打包（三标准任一即可）：**
1. **重复 3 次以上**：同一段 prompt 写过 3 次以上 = 打包信号。
2. **团队要对齐基线**：多人做同一类事、做法不一致，需要统一标准。
3. **经验要传承**：资深做法需要沉淀，供新人/他人复用。

**先别急（三暂缓情形）：**
1. 偶尔一次的特殊任务。
2. 还在探索期、做法未成型。
3. 内容每次都不同、无稳定结构。

> 过早打包 = 浪费时间；不打包的代价 = 质量飘忽、经验留不下、协作有壁垒。

**正反对照：**
- ✅ 正面："每周都要把多份数据周报汇总成统一格式，已手动做过 5 次" → 命中标准 1，值得打包。
- ❌ 反面："明天要给客户做一次性的特殊汇报" → 偶尔一次，先别急。

## 二、六要素判定细则（审核标准，缺一不可）

### 要素① 触发名
**判定要点**：动词 + 对象；触发句式明确固定（如"用 [X] 处理："），能一秒识别；description 写清何时触发。
- ✅ 正面：名称「周报汇总」，触发句式"用周报汇总处理：……"，动词+对象，一看就知道干什么。
- ❌ 反面：名称「小助手」「万能工具」—— 无动词无对象，不知道何时该喊它。
- ⚠️ 不足：名称合格但 description 未写触发场景，模型不知道何时调用。

### 要素② 输入要求
**判定要点**：写明必填 / 可选 / 形式（格式、长度、来源），从模糊到具体。
- ✅ 正面："必填：录音文字稿（≥500 字纯文本）；可选：与会人名单；形式：直接粘贴文本或给出文件路径。"
- ❌ 反面："给我会议内容就行" —— 无形式、无边界，输入参差不齐时输出必然飘忽。
- ⚠️ 不足：写了必填项但没写形式与边界（如未说明超长输入如何处理）。

### 要素③ 指令本体
**判定要点**：分步描述（1. 2. 3.），每步可执行、无歧义；复杂判断给出明确规则而非模糊要求。
- ✅ 正面："1. 通读全文，识别决策、待办、遗留三类内容；2. 待办逐条标注负责人与截止时间，原文未提的标「待确认」；3. 文末追加风险提示，须引用原文具体段落。"
- ❌ 反面："整理一下内容，弄得清楚点。" —— 不可执行，一千次运行一千种结果。
- ⚠️ 不足：有分步但缺关键判断规则（如"分不清归哪类时怎么办"未规定）。

### 要素④ 输出格式
**判定要点**：结构化、有模板或字段级约定（分几块、每块什么结构、长度限制），输出可校验。
- ✅ 正面："固定三段：决策（≤3 条，每条 ≤1 行）/ 待办（表格：事项｜负责人｜截止）/ 遗留（2-3 条，每条带一个待澄清问题）。"
- ❌ 反面："输出整理结果。" —— 无结构约定，输出格式无法校验。
- ⚠️ 不足：有结构但缺兜底（如某块无内容时未规定补「无」）。

### 要素⑤ 示例
**判定要点**：给出 good input → good output 的完整对照；正反面都给 —— 正面的供 follow，反面的供规避。示例是稳定性的最大杠杆：1 个好样本 ≈ 100 次稳定输出。
- ✅ 正面：一段真实感输入 + 对应的完美输出 + 一份"反面教材"（错误输出 + 错因注解，如"负责人脑补成主持人——违反「未提则标待确认」"）。
- ❌ 反面：无任何示例；或只写"输出要清晰美观"这类无法作为样本的描述。
- ⚠️ 不足：只有正面示例没有反面教材（稳定性打折扣）；或示例与输出格式约定不一致（自相矛盾，危害大于无示例）。

### 要素⑥ 校验规则
**判定要点**：清单化红线自检，输出前逐条过；规则可以尽可能多，覆盖已知翻车点。
- ✅ 正面："输出前自检：1. 三块结构齐全，缺块补「无」；2. 负责人/截止原文未提的已标「待确认」，未脑补；3. 弱承诺（"尽量""争取"）保留原语气，未升级为确定动作；4. 风险提示引用了原文段落。"
- ❌ 反面：无校验规则，出错全靠运气。
- ⚠️ 不足：校验只有一句"检查输出是否正确"——不可执行的伪校验。

## 三、六项质量特性判定细则

单字诀：**专 · 界 · 验 · 复 · 享 · 迭**。总口诀：**要素全，特性稳**。

| 特性 | 含义 | ✅ 正面 | ❌ 反面 |
|---|---|---|---|
| **专** | 单一职责 | 一个 skill 只做一件事（如只做周报汇总） | 一个 skill 又写周报又审合同又做翻译 |
| **界** | 输入边界清晰 | 输入要求具体到形式与边界 | "啥都能喂"，喂进去跑不动或乱跑 |
| **验** | 输出可校验 | 输出结构明确，能逐条核对对错 | 输出凭感觉，好坏无法判断 |
| **复** | 可复用 | 换一份同类素材照样稳定输出 | 只在作者演示的那一份素材上好使 |
| **享** | 可共享 | 描述自包含，他人拿来就会用 | 依赖作者脑中的隐含背景，别人用不了 |
| **迭** | 可迭代 | 规则分条清晰，改一处不牵全身 | 一整段纠缠的散文，改一处崩三处 |

## 四、检验方法与不稳定根因表

**检验一个 skill 是否优秀的实操方法：**
1. **真实素材对比测试**：同一份素材，随手 prompt 跑一次、skill 调用跑一次，对比稳定性与格式一致性。
2. **校验规则自检**：输出前按清单过结构（缺块补「无」）、未确认项（标「待确认」、不脑补）。
3. **迭代观察**：V1 → V2 → V3 收敛为健康，用一次改一次，3-5 轮稳定。

**不稳定根因定位表（按排查顺序）：**

| 症状 | 首要根因 | 对应要素 | 修法 |
|---|---|---|---|
| 输出时好时坏、格式飘忽 | 示例不够 | 要素⑤ 示例 | 补 good input → good output 完整对照，加反面教材 |
| 同类错误反复出现 | 校验没写 | 要素⑥ 校验规则 | 把该错误加成一条清单化红线 |
| 某些输入直接跑崩 | 输入边界未定义 | 要素② 输入要求 | 补必填/可选/形式与异常输入处理规则 |
| 输出结构缺斤短两 | 输出格式无兜底 | 要素④ 输出格式 | 规定缺块补「无」并加入校验规则 |
| 大家各自喊不动它 | 触发名/句式不清 | 要素① 触发名 | 改动词+对象，固定触发句式 |
| 什么都想干、什么都不稳 | 职责过多 | 特性「专」不足 | 拆成多个单一职责 skill |
