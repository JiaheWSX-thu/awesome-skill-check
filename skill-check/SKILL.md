# Skill Check —— Skill 质量审核门禁

## 用途

对任意 Skill（SKILL.md 及其捆绑资源）执行质量审核、选题评估或不稳定排查，输出结构化审核报告。审核依据为「六要素 · 六步骤」模型：**六要素是审核标准（缺一不可），六步骤是审核流程**；另有六项质量特性（**专 · 界 · 验 · 复 · 享 · 迭**）作为进阶评估。总口诀：**要素全，特性稳**。标准全文与正反面例子见 references/quality-standards.md，核对时按需查阅。

**创建即审**：用户创建新 Skill 时，初稿完成后即触发本技能做场景 1 全面审核，合格后方可使用或发布——把问题拦在上线前。

## 场景判定（先判定，再执行）

| 场景 | 用户表现 | 执行范围 |
|---|---|---|
| 1. 全面审核（默认） | 检查 / 评审 / 审核 / review 一个已存在的 skill；**或新 skill 初稿完成** | 完整六步骤 |
| 2. 选题评估 | 还没有 skill，只有一个想法，问"值不值得做" | 只做第 2 步，结论三档：值得打包 / 先别急 / 需要补信息 |
| 3. 不稳定排查 | skill 能跑但输出飘忽、时好时坏 | 完整流程，重点查第 3 步的「示例」与「校验规则」+ 根因定位 |

## 输入要求

- **必填（场景 1/3）**：被审 skill 的目录路径、SKILL.md 路径或全文内容，三者任一。
- **强烈建议（场景 3）**：一次真实运行的「输入 + 实际输出 + 期望输出」样本。无样本则降级为静态审核，报告中必须标注"未做动态排查"。
- **可选**：捆绑资源清单、使用背景（个人自用 / 团队共享，影响「享」特性权重）、审核侧重点。
- **场景 2 只需**：一句话描述要做的事情 + 大致使用频率。

## 反问规则（要不要反问用户）

默认不反问。仅以下三种情形反问，一次不超过 3 个问题，每个问题附带默认假设；反问得到的信息直接落入报告与修复建议，不白问：

1. **未提供任何 skill 路径或内容** → 反问获取（无默认假设，阻塞等待）。
2. **场景 3 但无真实运行样本** → 反问："最近一次坏输出的输入是什么？实际输出什么样？期望什么样？"（用户给不出则降级为静态审核并在报告标注，不阻塞）。
3. **修复方案涉及内容取舍**（如是否拆分 skill、是否删改用户原有内容）→ 先给带假设的建议，标注「待确认」，不阻塞报告。

除上述情形外，背景信息缺失时在报告中写明默认假设（如"默认按团队共享标准审核"）并继续，不逐项追问。

## 审核工作流（六步骤）

1. **定位读取**：读取 SKILL.md 全文与 YAML frontmatter（核对 name、description、agent_created 是否齐全合规），列出目录结构，确认声明的 scripts/ references/ assets/ 真实存在且在正文中被引用。
2. **选题评估**：对照三标准（重复 3 次以上 / 团队要对齐基线 / 经验要传承，任一即可）与三暂缓情形（偶尔一次 / 探索期未成型 / 内容每次都不同）。已在用的 skill 此步从简；场景 2 此步为核心产出。
3. **六要素逐项审核**：触发名 → 输入要求 → 指令本体 → 输出格式 → 示例 → 校验规则，缺一即拦。每项判 ✅ / ⚠️ / ❌，并引用被审 skill 原文作为依据。判定标准与正反面例子见 references/quality-standards.md。
4. **质量特性评估**：专（单一职责）/ 界（输入边界清晰）/ 验（输出可校验）/ 复（可复用）/ 享（可共享）/ 迭（可迭代），逐项给结论与依据。
5. **输出报告**：按下述「输出格式」写审核报告。问题按 P0 / P1 / P2 分级，每条带可执行修法（改前 → 改后对照）；禁止只批评不给方案。
6. **迭代复检**：修复后重新审核，预期 V1 → V3、3-5 轮收敛。场景 3 的根因定位先查两件事：**示例不够 / 校验没写**，再查其余要素。

## 输出格式（审核报告）

报告使用 plain Markdown，结论先行，严格按以下结构输出：

```markdown
# Skill 审核报告：<skill-name>

## 结论
【优秀 / 合格 / 需修改后可用 / 不合格】—— 一句话理由。
（场景 2 替换为：【值得打包 / 先别急 / 需要补信息】—— 一句话理由。）

## 逐项核对表
| 检查项 | 判定 | 依据（引用原文） |
|---|---|---|
| 要素①触发名 | ✅/⚠️/❌ | …… |
| 要素②输入要求 | …… | …… |
| 要素③指令本体 | …… | …… |
| 要素④输出格式 | …… | …… |
| 要素⑤示例 | …… | …… |
| 要素⑥校验规则 | …… | …… |
| 特性：专 / 界 / 验 / 复 / 享 / 迭 | …… | …… |

## 问题清单（按严重度）
- **P0 阻断**（不修不能用）：位置 + 问题 + 依据 + 修法（改前 → 改后）
- **P1 质量**（不修不好用）：位置 + 问题 + 依据 + 修法（改前 → 改后）
- **P2 建议**（锦上添花）：位置 + 问题 + 依据 + 修法

## 修复建议
按 P0 → P1 → P2 顺序列出可直接执行的修改；涉及取舍的标注「待确认」。

## 根因定位（仅场景 3）
不稳定根因 → 对应要素缺口 → 修法。无真实样本时标注"未做动态排查"。

## 自检确认
确认本报告已通过下方全部校验规则，逐项打勾。
```

完整正反对照示例（含拦截修改与反问全过程剧本）见 references/review-examples.md，写报告前查阅。

## 校验规则（报告输出前逐条自检，红线）

1. 六要素 + 六项特性全覆盖，无缺项；被审 skill 本身缺某项时判 ❌ 而非从报告中省略。
2. 每个判定、每条问题必须引用被审 skill 原文作依据；依据不足处标「待确认」，不脑补、不凭空断言。
3. 每条问题必须附可执行修法；只批评不给方案的报告视为不合格，重写。
4. 判定标准一律以 references/quality-standards.md 为准，不自造标准。
5. 问题分级正确：缺要素、触发名失效、无输出格式 → P0；示例缺反面、校验规则过少 → P1；表述优化 → P2。
6. 报告结构完整（结论 / 核对表 / 问题清单 / 修复建议 / 自检确认），结论先行。
7. 审核过程不修改被审 skill 本体；仅当用户明确要求"直接修复"时才动手，且修复后复检一次。
