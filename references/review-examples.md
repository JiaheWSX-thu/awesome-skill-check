# 审核剧本与报告正反对照示例

审核有两个必须覆盖的时机，写报告前先确认自己处在哪一个：

| 时机 | 用户表现 | 对应场景 | 审核动作核心 |
|---|---|---|---|
| **时机一 · 创建时** | 刚写完 skill 初稿（创建即审） | 场景 1 全面审核 | 拦截 → 修改 → 复检放行 → 升华 |
| **时机二 · 使用中出现问题** | skill 在真实使用中输出飘忽、不好用 | 场景 3 不稳定排查 | 反问取样 → 根因 → 修复 → 固化升华 |

每个剧本都是完整流程示范。示例中的被审 skill 均为通用虚构示例。

---

## 剧本一【时机一 · 创建即审】：拦截 → 修改 → 放行 → 升华

**背景**：用户刚写完一个 skill 初稿，触发创建即审。

**被审 skill 初稿全文**：

```markdown
---
name: data-helper
description: 帮你处理数据
---
# 数据助手
把数据发给我，我帮你分析得清晰美观。
```

### 第一轮：审核与拦截

**审核动作（六步骤执行）**：

1. **定位读取**：frontmatter 有 name/description，无捆绑资源；description 过简。
2. **选题评估**：用户称"每周都要处理数据周报" → 命中"重复 3 次以上"，选题合格，继续。
3. **六要素审核**：逐项判 ❌（见下表），**门禁拦截：结论【不合格】，禁止使用/发布**。
4. **质量特性**：「专」⚠️（"处理数据"范围过宽）。
5. **输出报告**（关键节选）：

```markdown
# Skill 审核报告：data-helper

## 结论
【不合格】—— 六要素缺 5 项，结构不完整，已拦截，需按 P0 清单修改后复审。

## 逐项核对表
| 检查项 | 判定 | 依据（引用原文） |
|---|---|---|
| 要素①触发名 | ❌ | 名称「data-helper」无动词+对象；description 仅"帮你处理数据"，未写触发场景 |
| 要素②输入要求 | ❌ | 仅"把数据发给我"，未约定必填/可选/形式 |
| 要素③指令本体 | ❌ | 仅"我帮你分析"，无分步指令 |
| 要素④输出格式 | ❌ | "清晰美观"无法校验，无结构约定 |
| 要素⑤示例 | ❌ | 全文无任何示例 |
| 要素⑥校验规则 | ❌ | 全文无校验规则 |
| 特性·专 | ⚠️ | "处理数据"范围过宽，疑似多职责，待确认实际用途 |

## 问题清单
- **P0** 触发名失效 → 改前：`name: data-helper` / `description: 帮你处理数据`；
  改后：`name: data-weekly-summary` / `description: 汇总数据周报。当用户要求汇总周报、合并多份数据周报为统一格式时使用。触发词：汇总周报、周报合并、weekly summary。`
- **P0** 输入无要求 → 改前：`把数据发给我`；
  改后：`输入要求：必填——2 份以上数据周报（CSV 文件路径或直接粘贴的表格文本）；可选——汇总口径（默认按周）；单份超过 5000 行时先让用户确认抽样方式。`
- **P0** 指令不可执行 → 改前：`我帮你分析`；
  改后：`1. 读取全部周报，校验列名一致，不一致的列列入「口径差异」；2. 按统一模板合并为一张总表；3. 标注各周报独有的指标；4. 输出汇总结果。`
- **P0** 输出无格式 → 改前：`清晰美观`；
  改后：`固定三段：汇总总表（Markdown 表格）/ 口径差异（逐条列出，无差异则写「无」）/ 独有指标（按周报分组）。`
- **P0** 无示例 → 补一组：2 份各 10 行的示例周报文本 → 对应的完整汇总输出。
- **P1** 无反面示例、无校验规则 → 示例追加 1 份反面教材（如"口径差异漏标"）；补校验规则：`输出前自检：1. 三段齐全，缺段补「无」；2. 所有输入周报均已入表，无遗漏；3. 未擅自修改原始数值。`

## 修复建议
按上述改后文本逐条替换，替换后重新提交审核。

## 自检确认
六要素+六项特性已全覆盖；每条问题均附原文依据与改后文本；分级符合规则 5。
```

### 第二轮：修改与放行

用户按 P0/P1 清单逐条修改，重新提交 → 六要素全部 ✅、特性无 ❌ → 结论【合格】，**门禁放行，skill 可以投入使用**。

### 第三轮：升华（合格 → 优秀）

放行不是终点。审核方基于标准主动给出一轮 P2 升华建议（每条仍落在具体要素上，可执行、不堆口号）：

```markdown
## 升华建议（P2，可选）
1. 要素⑤：示例库再补 1 个"超长周报（>5000 行）"的边界样本，验证抽样确认规则真的生效——强化特性「界」。
2. 要素①：收集团队对这件事的口语叫法（如"并周报"）补进触发词——强化触发命中率。
3. 要素⑥：把本次 P0 踩过的坑（无输出格式、无示例）写成一条作者自查项："新 skill 提交前先自查六要素齐全"——同类问题不复发。
```

用户采纳 → V2 复审 → 结论【优秀】。**剧本要点**：创建即审 = 先把不合格的拦在上线前（P0 改前→改后可照抄），放行后再推一轮小幅升华；一轮只推进一步，每步都可执行——V1 能用、V2 好用，不追求一步登天。

## 剧本二【时机二 · 使用中求诊】：反问 → 根因 → 修复 → 固化

**背景**：skill 已投入使用，用户反馈："我那个周报汇总的 skill 不好用，输出时好时坏。"只给了 skill 路径，没给任何运行样本。

### 第一步：判定缺样本，触发反问

（反问规则第 2 条，一次 3 问，附默认假设与降级路径）

> 要做动态排查需要一次真实失败样本，请补充三个信息：
> 1. 最近一次坏输出，你喂给它的输入是什么？（贴原文或给文件路径）
> 2. 它实际输出了什么样？（贴关键片段即可）
> 3. 你期望的输出应该长什么样？
> 如果你现在拿不到样本，我可以先按静态审核出报告并标注"未做动态排查"。

### 第二步：用户回答

> 输入是三份周报粘贴文本；实际输出里"口径差异"这一段时而有时而没有；期望每段都有，没有内容就写"无"。

### 第三步：反问信息落入排查

对照被审 skill 原文，发现要素④输出格式只写了"输出三段结构"，**没有规定"无内容段落如何兜底"**，且要素⑥校验规则不含"三段齐全"检查 → 根因定位：**输出格式无兜底 + 校验漏项**。

### 第四步：报告根因定位与修复（节选）

```markdown
## 根因定位（场景 3）
症状：「口径差异」段时有时无（用户实测样本确认）
根因：要素④输出格式缺"缺块补「无」"兜底规则；要素⑥校验规则未覆盖"三段齐全"
修法：1. 输出格式补写「任一段无内容时输出『无』，不得省略该段」；
      2. 校验规则新增「三段齐全，缺段补『无』」；
      3. 示例库追加 1 个"无口径差异"的样本（good input → 含「无」段的 good output）。
```

### 第五步：固化（修复之上再升华一步）

问题修好后，把这次的症状变成长期防线，让 skill 从"修好"变成"更稳"：

```markdown
## 固化建议（防止复发）
1. 把本次故障样本（三份周报 → 缺段输出）收进示例库当反面教材，标注错因——以后同类漂移有对照。
2. 建议用户约定：每次真实使用发现新问题，顺手把样本带回审核，纳入示例或校验——skill 随使用持续变稳。
```

演进路径：**V1（能跑）→ V2（本次修复，不漂移）→ V3（故障样本入库，稳）**。

**剧本要点**：使用中出问题，先反问拿真实样本，不硬猜；用户答案逐条落进根因与修法——不白问；修复后把症状固化为示例与校验，同样的问题只犯一次。

---

## 剧本三：选题评估中的反问（场景 2）

- 用户："我想给写作做个 skill，你看值不值？"
- 审核动作：信息不足（频率未知）→ 按规则反问 1 问带默认假设："这件事过去一个月你做过几次？（默认假设：≥3 次则值得打包）"
- 用户答："就上周写过一次。" → 结论【先别急】——次数未达 3 次且内容每次不同，建议再积累两次同类型素材后再来评估。

---

## ❌ 不合格审核报告（典型毛病，逐条规避）

```markdown
这个 skill 写得不太好。触发名不行，输入也没写清楚，示例最好加一个，
整体比较乱，建议重新写一下。另外校验规则也可以考虑加上。
```

**毛病清单**：
1. **无结论分档**——没有【优秀/合格/需修改后可用/不合格】的明确判定，也没有"拦截/放行"动作。
2. **无原文依据**——"触发名不行"却没引用原文、没说为什么不行，属于凭空断言。
3. **只批评不给修法**——"建议重新写一下"不可执行；每条问题必须给「改前 → 改后」。
4. **漏项**——未覆盖六要素 + 六项特性（少了哪几项？没说）。
5. **无分级**——P0 阻断与 P2 建议混在一堆，用户不知道先修什么。
6. **语气 hedge 过多**——"最好""可以考虑"削弱指令性；报告用确定性表述。
7. **只有把关没有升华**——合格的报告在放行/修复后，还给一条可执行的下一步建议，让 skill 一轮比一轮好。
